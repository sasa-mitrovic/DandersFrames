# CI/CD & Releases

Build pipeline, release flow, and version management. Uses GitHub Actions + BigWigs Packager for CurseForge uploads.

**Workflow files:** `.github/workflows/release.yml`, `.github/workflows/nightly-beta.yml`
**Packaging config:** `.pkgmeta`
**Changelog generator:** `generate_changelog.sh`

## Build Guides

For step-by-step instructions on each release type, see:

| Channel | Guide | Trigger |
|---------|-------|---------|
| Alpha | [build-alpha.md](build-alpha.md) | Manual tag (`v4.0.7-alpha.N`) |
| Beta | [build-beta.md](build-beta.md) | Automatic nightly or manual |
| Stable | [build-stable.md](build-stable.md) | Manual tag (`v4.0.7`) |

## Release Flow Overview

- **Tag `v4.0.7-alpha.1`** --> **alpha** build uploaded to CurseForge + GitHub Release
- **Nightly (3 AM UTC)** --> automatic **beta** build if new commits exist --> CurseForge + GitHub Release
- **Tag `v4.0.7`** (no suffix) --> **stable release** to CurseForge + GitHub Release
- **Manual trigger** --> Actions --> Package and Release --> Run workflow
- **Push to `main`** --> triggers CI, but build guard skips docs-only changes. **Use explicit alpha tags instead.**

## Release Channel Gating

Features not ready for stable release can be gated:
```lua
if DF.RELEASE_CHANNEL == "release" then
    -- Show "Coming Soon" or skip feature
    return
end
```
`DF.RELEASE_CHANNEL` is `"alpha"`, `"beta"`, or `"release"` (set in `Changelog.lua` by CI). Gate both the UI (in `BuildPage`) and any event handlers or runtime logic.

## Important Rules

- **`Changelog.lua` is auto-generated by CI.** Never edit it manually. Locally it contains a dev stub; CI overwrites it with real version/changelog data during builds. You can run `bash generate_changelog.sh` locally to regenerate it for in-game testing, but don't commit the result.
- **Always regenerate the local `Changelog.lua` before finishing a session** so the in-game changelog stays up to date for local testing. Run `bash generate_changelog.sh` then restore CHANGELOG.md with `git checkout -- CHANGELOG.md DandersFrames.toc` (the script adds an "Unreleased" section as a side effect). Never commit the generated `Changelog.lua` or the modified `DandersFrames.toc`.
- **`CHANGELOG.md` is human-maintained.** This is the source of truth for what players see in the in-game changelog.
- **Version numbers come from git tags**, not from `CHANGELOG.md`. Bumping the version in `CHANGELOG.md` alone does NOT change the build version — you must create a tag.
- **Add `[skip ci]` to commit messages** for changes that should not trigger a build. The build guard also auto-skips commits that only touch `.md`, `.sh`, `.github/*`, `.gitignore`, or `.pkgmeta` files.
- **Alpha tags must be created manually.** Nightly beta tags are auto-created by CI. Stable release tags are manual.

## Version Variables

```lua
DF.ADDON_VERSION    -- Set by CI (e.g. "v4.0.6", "v4.0.5-alpha.3"), "dev" locally
DF.VERSION          -- From TOC metadata (GetAddOnMetadata), "@project-version@" locally
DF.RELEASE_CHANNEL  -- "alpha", "beta", or "release"
DF.CHANGELOG_TEXT   -- Full changelog text for in-game display
DF.BUILD_DATE       -- ISO 8601 build timestamp
```

## Tool Notes

### GitHub CLI
`gh` is installed at `C:\Program Files\GitHub CLI`. In the bash shell used by Claude Code, it needs to be added to PATH:
```bash
export PATH="$PATH:/c/Program Files/GitHub CLI"
```

### TOC Metadata
- `X-Curse-Project-ID: 1389690` — CurseForge project identifier
- `X-Wago-ID: RNL9B46o` — Wago Addons project identifier (required for Wago update detection)
- `AllowAddOnTableAccess: 1` — enables external addons to access the `DF` table (required for the Wago API)
